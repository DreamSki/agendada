# 🎯 场景 2：笔记操作测试指南

## 测试目标

测量**快速创建/编辑/切换笔记**时的性能，发现：
- `snapshot()` 执行时间
- `publishChange()` 调用频率
- `persist()` 是否阻塞主线程
- 内存是否增长

---

## 📋 测试步骤（10 分钟）

### 准备

1. **打开 Activity Monitor**（保持打开）
2. **在 Instruments 中新建 Time Profiler**：
   - File → New → Time Profiler
   - Target: `.build/debug/Agendada`

### 操作序列（重复 20 次）

在 Agendada 中执行以下操作 20 次：

```
1. 点击 + 按钮（新建笔记）
2. 输入标题 "Test 1", "Test 2", ... "Test 20"
3. 输入正文 "test content"
4. 按 ESC 键（退出编辑）
5. 点击下一笔记（或保持当前位置）
```

**计时**：
- 建议每次操作间隔 2-3 秒
- 总计约 2-3 分钟

---

## 🔍 在 Instruments 中操作

### 1. 启动录制

- 点击红色录制按钮 ●
- 立即切换到 Agendada 开始操作

### 2. 执行 20 次笔记操作

- 快速但稳定地执行
- 如果中途卡顿，记录卡顿的位置

### 3. 停止录制

- 完成 20 次后，点击停止 ⏹
- 保存：`PerformanceResults/notes_operations.trace`

## ⚠️ 重点检查

### 检查 1：主线程阻塞

如果出现卡顿，在 Call Tree 中：
1. 查看 **"Running Time"** 最长的函数
2. 确认是否在主线程
3. 记录函数名和耗时

### 检查 2：内存增长

操作前后对比：
- 操作前内存：127 MB
- 操作后内存：___ MB
- 增长：___ MB

如果增长 > 50 MB，可能有问题。

### 检查 3：视图刷新

观察：
- 每次输入标题时，整个笔记列表是否闪烁？
- 切换笔记时，是否有明显延迟？

---

## 📝 预期问题（基于代码审查）

根据之前的分析，你应该会发现：

### 🔴 Critical（必须发现）

1. **`snapshot()` 调用**
   - 应该出现约 20 次（或更少，因为 debounce）
   - 如果每次操作都出现 → persistSoon 没有生效
   - 如果执行时间 > 50ms → 主线程阻塞

2. **`publishChange()` 频繁调用**
   - 应该出现约 60 次（每个操作 3 次：新建、标题、正文）
   - 如果远超这个数 → 有级联更新

### ⚠️ Warning（应该发现）

1. **`filteredNotes()` 被频繁调用**
   - 每次状态变化都可能触发
   - 如果缓存失效频繁 → 会有大量调用

## 💡 快速判断

测试完成后，如果：

- ✅ **CPU < 30%, Memory < 150 MB, 无明显卡顿**
  → 基本性能良好，可以继续场景 3

- ⚠️ **CPU > 50%, 或有卡顿**
  → 有性能问题，需要深入分析 Call Tree

- 🔴 **CPU > 80%, 或严重卡顿**
  → 有严重性能问题，必须立即修复
